🏗️ Overall Architecture Flow
User → Controller → Service → Repository → MySQL DB
                ↑
         JWT Filter (validates token on every request)

👤 1. User Registration & Login Flow
User sends POST /api/auth/register
        ↓
AuthController receives RegisterRequest
        ↓
AuthServiceImpl checks: does email already exist?
        ↓ NO                    ↓ YES
Save User to DB          throw 400 "Email already registered"
(password BCrypt hashed)
        ↓
UUID auto-generated by Hibernate (@UuidGenerator)
        ↓
JwtUtil.generateToken(email) → creates JWT (24hr expiry)
        ↓
Return AuthResponse { token, userUuid, name, email, role }
On every subsequent request:
Request comes in with "Authorization: Bearer <token>"
        ↓
JwtAuthenticationFilter intercepts
        ↓
JwtUtil.extractEmail(token) → gets email from token
        ↓
UserRepository.findByEmail(email) → loads User from DB
        ↓
Sets SecurityContext → Spring knows who is logged in
        ↓
@AuthenticationPrincipal User currentUser → injected into controller

👥 2. Group Creation Flow
Alice calls POST /api/groups  { name: "Goa Trip" }
        ↓
GroupController → GroupServiceImpl.createGroup()
        ↓
Save Group entity to expense_groups table
(uuid auto-generated, createdBy = Alice)
        ↓
Auto-add Alice as first GroupMember
(group_members table: group_id + user_id)
        ↓
Return GroupResponse { uuid, name, members: [Alice] }
Add Bob to group:
Alice calls POST /api/groups/{groupUuid}/members { userUuid: bobUuid }
        ↓
Check: is currentUser (Alice) the group creator? YES → proceed
        ↓
Find Bob by UUID in users table
        ↓
Check: is Bob already a member? NO → proceed
        ↓
Save new GroupMember { group, user: Bob }
        ↓
Return updated GroupResponse { members: [Alice, Bob] }

🧾 3. Expense Flow (most complex part)
Alice calls POST /api/expenses
{
  description: "Hotel",
  amount: 3000,
  splitType: EQUAL,
  groupUuid: "...",
  paidByUserUuid: aliceUuid
}
        ↓
Find Group by UUID → verify exists
        ↓
Find paidBy User by UUID → verify is group member
        ↓
Save Expense entity to expenses table
        ↓
Load all GroupMembers of the group [Alice, Bob, Charlie]
        ↓
        ┌─────────────────────────────────────┐
        │         SPLIT CALCULATION           │
        │                                     │
        │  EQUAL:      3000 / 3 = 1000 each  │
        │                                     │
        │  EXACT:      Alice=600, Bob=500,    │
        │              Charlie=400            │
        │              (must sum to 3000)     │
        │                                     │
        │  PERCENTAGE: Alice=40% → 1200       │
        │              Bob=35%  → 1400        │
        │              Charlie=25% → 1000     │
        │              (must sum to 100%)     │
        └─────────────────────────────────────┘
        ↓
Save ExpenseSplit rows (one per member)
expense_splits table:
  expense_id | user_id | share_amount | settled
  exp1       | Alice   | 1000         | false
  exp1       | Bob     | 1000         | false
  exp1       | Charlie | 1000         | false
        ↓
Return ExpenseResponse with splits[]

📊 4. Balance Calculation Flow
Alice calls GET /api/balances/group/{groupUuid}
        ↓
Load all Expenses in the group
Load all ExpenseSplits for the group
        ↓
Calculate Alice's totals:
  yourTotalPaid  = sum of expenses where paidBy = Alice
  yourTotalShare = sum of ExpenseSplits where user = Alice
  netBalance     = yourTotalPaid - yourTotalShare
        ↓
Per-person balance calculation:
  For each ExpenseSplit in group:
    paidBy ≠ splitUser → means splitUser OWES paidBy

    If paidBy = Alice → someone owes Alice (+)
    If splitUser = Alice → Alice owes someone (-)
        ↓
Build netBalanceMap { userId → amount }
  positive = they owe Alice
  negative = Alice owes them
        ↓
Adjust for completed Settlements:
  Bob paid Alice 1400 → Bob's debt reduces by 1400
        ↓
Return BalanceSummaryResponse:
{
  totalExpenses: 8500,
  yourTotalPaid: 3000,       ← Alice paid hotel
  yourTotalShare: 2833.33,   ← Alice's fair share
  netBalance: 166.67,        ← Alice is slightly owed
  balances: [
    { withUser: Bob,     amount: X, status: "THEY OWE YOU" },
    { withUser: Charlie, amount: Y, status: "THEY OWE YOU" }
  ]
}

💸 5. Settlement Flow
Bob calls POST /api/settlements
{
  payerUuid: bobUuid,        ← Bob is paying
  receiverUuid: aliceUuid,   ← Alice is receiving
  groupUuid: "...",
  amount: 1400,
  paymentMethod: UPI,
  transactionRef: "UPI/TXN/..."
}
        ↓
Validate: currentUser (Bob) == payer → YES, proceed
        ↓
Validate: both payer & receiver are group members
        ↓
Save Settlement {
  payer: Bob,
  receiver: Alice,
  status: COMPLETED,
  settledAt: now()
}
        ↓
Save Payment {
  settlement: above,
  amount: 1400,
  paymentMethod: UPI,
  transactionRef: "UPI/TXN/..."
}
        ↓
Return SettlementResponse { uuid, payerName, receiverName, amount, status: COMPLETED }
        ↓
Next time GET /balances is called →
  Settlements are factored in →
  Bob's debt is reduced by 1400

🗄️ 6. Database Table Relationships
users
  id | uuid | name | email | password | role
  ─────────────────────────────────────────────
   1 | a1.. | Alice | alice@.. | hash | USER
   2 | b2.. | Bob   | bob@..   | hash | USER
   3 | c3.. | Charlie | ..     | hash | USER

expense_groups
  id | uuid | name       | created_by(FK→users)
  ─────────────────────────────────────────────
   1 | g1.. | Goa Trip   | 1 (Alice)

group_members
  id | group_id(FK) | user_id(FK) | joined_at
  ─────────────────────────────────────────────
   1 | 1            | 1 (Alice)   | 2025-01-15
   2 | 1            | 2 (Bob)     | 2025-01-15
   3 | 1            | 3 (Charlie) | 2025-01-15

expenses
  id | uuid | description | amount | split_type | paid_by(FK) | group_id(FK)
  ──────────────────────────────────────────────────────────────────────────
   1 | e1.. | Hotel       | 3000   | EQUAL      | 1 (Alice)   | 1
   2 | e2.. | Dinner      | 1500   | EXACT      | 2 (Bob)     | 1
   3 | e3.. | Car         | 4000   | PERCENTAGE | 3 (Charlie) | 1

expense_splits
  id | expense_id(FK) | user_id(FK) | share_amount | settled
  ────────────────────────────────────────────────────────────
   1 | 1 (Hotel)      | 1 (Alice)   | 1000         | false
   2 | 1 (Hotel)      | 2 (Bob)     | 1000         | false
   3 | 1 (Hotel)      | 3 (Charlie) | 1000         | false
   4 | 2 (Dinner)     | 1 (Alice)   | 600          | false
   5 | 2 (Dinner)     | 2 (Bob)     | 500          | false
   6 | 2 (Dinner)     | 3 (Charlie) | 400          | false

settlements
  id | uuid | payer(FK) | receiver(FK) | group(FK) | amount | status
  ──────────────────────────────────────────────────────────────────
   1 | s1.. | 2 (Bob)   | 1 (Alice)    | 1         | 1400   | COMPLETED

payments
  id | uuid | settlement_id(FK) | amount | payment_method | transaction_ref
  ──────────────────────────────────────────────────────────────────────────
   1 | p1.. | 1                 | 1400   | UPI            | UPI/TXN/...

🔐 7. Security Flow Summary
Public (no token needed):
  POST /api/auth/register
  POST /api/auth/login

Protected (token required):
  Everything else → JwtAuthenticationFilter validates token first

Role-based:
  GET /api/users → ADMIN only (@PreAuthorize)

Ownership checks (done in service layer):
  Add/remove member  → only group creator
  Delete group       → only group creator
  Delete expense     → only payer or group creator
  Create settlement  → only the payer themselves

🔄 Complete Request Lifecycle (one full example)
Bob sends: POST /api/expenses  +  Bearer token
           ↓
1. JwtAuthenticationFilter
   → extracts token → finds Bob in DB → sets SecurityContext

2. Spring routes to ExpenseController.addExpense()
   → @AuthenticationPrincipal injects Bob as currentUser

3. ExpenseController calls expenseService.addExpense(request, bob)

4. ExpenseServiceImpl:
   → finds Group by groupUuid
   → finds paidBy User by paidByUserUuid
   → verifies paidBy is group member
   → saves Expense to DB
   → calculates splits based on EQUAL/EXACT/PERCENTAGE
   → saves ExpenseSplit rows to DB
   → maps to ExpenseResponse

5. Controller wraps in ApiResponse.success(...)

6. GlobalExceptionHandler catches any error
   → returns proper 400/403/404/500 with message

7. Response sent back as JSON
