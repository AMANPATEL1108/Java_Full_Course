🚀 0. Big Picture (Internal View)

You’re not just “deploying an app.” You’re building a layered system:

Spring Boot → creates a runnable Java process
Docker → wraps that process with OS + dependencies
Kubernetes → orchestrates many such containers across machines

Think of it like this:

👉 JAR = program
👉 Image = packaged environment
👉 Container = running instance
👉 Kubernetes = distributed operating system

🧱 1. Maven Build (What happens internally)
mvn clean package
Inside Maven:
Reads pom.xml
Downloads dependencies into local .m2 repo
Compiles .java → .class
Runs test lifecycle
Uses Spring Boot plugin to:
Bundle dependencies into one JAR (fat JAR)
Add a custom classloader
Inside the JAR:
/BOOT-INF/classes     → your code
/BOOT-INF/lib         → dependencies
/org/springframework/boot/loader → launcher

👉 When you run:

java -jar app.jar

Spring Boot:

Starts embedded server (Tomcat/Netty)
Creates IoC container
Wires beans
Opens port (8080)
🐳 2. Docker (What really happens)
Image build process:
docker build -t my-app .

Docker uses Union File System (layers):

Layer 1 → OS (Debian slim)
Layer 2 → JDK
Layer 3 → your JAR

👉 Each layer is immutable & cached.

Container runtime:

When you run container:

Uses Linux kernel features:
Namespaces → isolation (PID, network, filesystem)
cgroups → resource limits (CPU, RAM)

👉 Important:
Containers are not VMs — they share host kernel.

📦 3. Registry (Why push?)

When you do:

docker push my-app

Internally:

Image layers are uploaded separately
Kubernetes later pulls these layers

👉 Optimization:
If node already has base layers → only new layers downloaded

☸️ 4. Kubernetes Architecture (Core Internal Components)

Kubernetes is a control plane + worker nodes

Control Plane:
API Server → entry point (kubectl talks here)
etcd → key-value DB (stores cluster state)
Controller Manager → ensures desired state
Scheduler → assigns Pods to nodes
Worker Node:
Kubelet → talks to API server, runs Pods
Container Runtime (containerd/Docker)
Kube-proxy → networking
📦 5. Deployment (What really happens)

When you apply YAML:

kubectl apply -f deployment.yaml
Internally:
YAML → sent to API Server
Stored in etcd
Deployment Controller wakes up:
Creates a ReplicaSet
ReplicaSet:
Ensures desired number of Pods
Pod creation:
Scheduler picks a node
Kubelet on that node:
Pulls image
Starts container

👉 This is a reconciliation loop:
Kubernetes constantly checks:

“Actual state == Desired state?”

🔁 6. Self-Healing Mechanism

If a Pod crashes:

Kubelet detects failure
Reports to API server
ReplicaSet creates a new Pod

👉 You never restart manually — system heals itself.

🌐 7. Networking (Most misunderstood part)

Every Pod gets:

Unique IP
Internal DNS name
Service abstraction:
type: NodePort

Internally:

kube-proxy creates iptables rules
Traffic routing:
NodeIP:30007
   ↓
Service (virtual IP)
   ↓
Random Pod (load balanced)

👉 Load balancing = round-robin at network layer

🔍 8. Labels & Selectors (Glue of Kubernetes)

This is critical:

labels:
  app: springboot

Used by:

Deployment → to manage Pods
Service → to route traffic

👉 No labels = nothing connects

⚙️ 9. Probes (How K8s decides app health)

Example:

livenessProbe:

Internally:

Kubelet periodically calls:

/actuator/health
If fails:
Liveness fails → container restarted
Readiness fails → removed from traffic

👉 This is how zero-downtime works.

⚠️ 10. Why “localhost” breaks in Kubernetes

Inside container:

localhost = same container only

So:

❌ localhost:3306 → wrong
✅ mysql-service:3306 → correct

👉 Kubernetes DNS resolves service names automatically.

🧠 The Core Idea (Everything boils down to this)

You describe desired state, and Kubernetes continuously forces reality to match it.

Everything else is just machinery to make that happen.

🧱 1. Build Stage — Turning Code into a Runnable Unit

When you run mvn package, you’re not just compiling code—you’re creating a self-contained runtime.

Your .java → .class
Dependencies bundled into one fat JAR
Spring Boot adds a custom launcher

When you run:

java -jar app.jar

Internally:

JVM starts
Spring Boot bootstraps an IoC container
Embedded server (Tomcat/Netty) starts
Beans are wired
Port opens (8080)

👉 At this point, you have a single-process application.

🐳 2. Docker — Making It Portable & Isolated

Docker takes that JAR and wraps it with:

OS layer (minimal Linux)
JDK
Your app

Built as immutable layers:

OS → JDK → App

When container runs:

Uses namespaces → isolation (network, PID, FS)
Uses cgroups → resource limits

👉 Key truth: Containers are just isolated processes sharing the host kernel, not full machines.

📦 3. Registry — Distribution Layer

When you docker push:

Layers are uploaded separately
Nodes later pull only missing layers

👉 This makes deployments fast and bandwidth-efficient.

☸️ 4. Kubernetes — The Brain (Control System)

Kubernetes is basically a distributed control loop system.

Control Plane Components:
API Server → entry point
etcd → stores desired + actual state
Scheduler → decides where things run
Controllers → enforce desired state
Worker Node:
Kubelet → executes instructions
Container runtime → runs containers
Kube-proxy → handles networking

👉 Important: Kubernetes doesn’t “run apps”—it manages state.

📦 5. Deployment Flow — What Actually Happens

When you run:

kubectl apply -f deployment.yaml

Internally:

YAML → API Server
Stored in etcd
Deployment Controller wakes up
Creates a ReplicaSet
ReplicaSet creates Pods
Scheduler assigns Pods to nodes
Kubelet pulls image & starts container

👉 This is a reconciliation loop:

Desired: 3 Pods
Actual: 2 Pods
→ Create 1 more
🔁 6. Self-Healing (Automatic Recovery)

If a container crashes:

Kubelet detects it
Reports to API Server
ReplicaSet creates a new Pod

👉 You don’t fix failures—Kubernetes replaces them.

🌐 7. Networking — How Traffic Reaches Your App

Each Pod:

Gets its own IP
Is ephemeral (can die anytime)

So Kubernetes introduces a Service:

Stable virtual IP
Load balances across Pods

Flow:

User → NodeIP:Port
     → Service
     → Pod (chosen via iptables rules)

👉 kube-proxy sets up iptables, not a traditional load balancer.

🔍 8. Labels — The Hidden Glue

Everything connects using labels:

labels:
  app: springboot
Deployment → manages Pods with label
Service → routes traffic to label

👉 No labels = no connection = nothing works.

⚙️ 9. Health Checks (Probes)

Kubernetes doesn’t trust your app—it verifies it.

Liveness probe → “Should I restart this?”
Readiness probe → “Should I send traffic?”

Example:

/actuator/health

👉 This is how Kubernetes achieves zero downtime deployments.

⚠️ 10. The “localhost” Trap

Inside a container:

localhost = this container only

So:

❌ localhost:3306
✅ mysql-service:3306

👉 Kubernetes has built-in DNS for service discovery.

🔥 End-to-End Flow (Compressed)
Code → JAR → Docker Image → Registry → Kubernetes

User Request:
Browser
 → NodePort
 → Service (virtual IP)
 → Pod
 → Container
 → Spring Boot
 → Response
🚀 Final Mental Model (Refined)
Spring Boot → creates a runnable process
Docker → makes it portable & isolated
Kubernetes → ensures it runs correctly, repeatedly, and at scale

👉 The real power is this loop:

Observe → Compare → Fix
