🚀 0. Big Picture (What You’re Actually Doing)

You’re converting this:

👉 Spring Boot app → runs on your laptop

Into this:

👉 Container → runs anywhere
👉 Kubernetes → manages containers at scale

So the flow is:

Code → JAR → Docker Image → Container → Kubernetes Pod → Service → Public Access

🧱 1. Build the Spring Boot Application
Command
mvn clean package
What actually happens?
Compiles Java code
Runs tests
Packages everything into a fat JAR
Includes embedded server (Tomcat/Netty)

👉 Output:

target/my-app.jar

This JAR is self-contained — you don’t need Tomcat separately.

🐳 2. Dockerizing the Application
Dockerfile
FROM openjdk:17-jdk-slim
COPY target/my-app.jar app.jar
ENTRYPOINT ["java", "-jar", "/app.jar"]
Deep explanation
FROM openjdk:17-jdk-slim
→ Base OS + Java runtime
COPY
→ Moves your JAR into container filesystem
ENTRYPOINT
→ Defines what runs when container starts

👉 When container runs:

java -jar /app.jar
Build Image
docker build -t your-dockerhub-username/my-app:1.0 .

👉 This creates a layered image:

OS layer
Java layer
App layer
Push to Registry
docker push your-dockerhub-username/my-app:1.0
Why this matters

Kubernetes cannot access your local machine
It pulls images from a registry like:

Docker Hub
AWS ECR
GCP Artifact Registry
☸️ 3. Kubernetes Core Concepts (Important Before YAML)
Pod

Smallest unit → runs 1+ containers

Deployment

Manages Pods:

scaling
updates
self-healing
Service

Exposes Pods:

internal or external traffic
📦 4. Deployment YAML (Deep Dive)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: springboot-deployment
Spec Section
spec:
  replicas: 2

👉 Runs 2 identical Pods

Load balancing
High availability
Selector
selector:
  matchLabels:
    app: springboot

👉 Tells Kubernetes:
“Manage Pods with this label”

Template (Pod blueprint)
template:
  metadata:
    labels:
      app: springboot

👉 Labels connect:

Deployment ↔ Pods
Service ↔ Pods
Container Definition
containers:
- name: springboot-container
  image: your-dockerhub-username/my-app:1.0
  ports:
    - containerPort: 8080

👉 Kubernetes:

Pulls image
Starts container
Exposes port internally
🌐 5. Service YAML (Networking Explained)
kind: Service
type: NodePort
What is NodePort?
Opens port on every node
Maps external → internal
ports:
  - port: 80
    targetPort: 8080
    nodePort: 30007
Flow:
Browser → NodeIP:30007 → Service → Pod:8080
⚙️ 6. Deploy to Kubernetes
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml
What happens internally?
Deployment created
ReplicaSet created
Pods created
Scheduler assigns Pods to nodes
Kubelet runs containers
🔍 7. Verify Deployment
kubectl get pods
kubectl get services
Debugging commands
kubectl describe pod <pod-name>
kubectl logs <pod-name>
🌍 8. Access Application
Minikube
minikube service springboot-service
Or manually:
http://<node-ip>:30007
🧠 9. Production-Level Improvements (Important)
🔹 Config Externalization

Instead of hardcoding DB:

spring.datasource.url=${DB_URL}
Use ConfigMap:
kind: ConfigMap
🔹 Secrets (for passwords)
kind: Secret
🔹 Health Checks

Add dependency:

spring-boot-starter-actuator
Why?

Kubernetes needs to know:

Is app alive?
Is app ready?
🔹 Probes
livenessProbe:
  httpGet:
    path: /actuator/health
    port: 8080
Difference:
Liveness → restart if broken
Readiness → stop traffic if not ready
🔹 Resource Limits
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"
    cpu: "500m"

👉 Prevents one app from killing the node

⚠️ 10. Real-World Mistakes Explained
❌ “Works locally but not in K8s”

Cause:

Using localhost for DB

👉 In Kubernetes:

Services talk via DNS:
mysql-service:3306
❌ ImagePullBackOff

Cause:

Wrong image name
Not pushed to registry
❌ CrashLoopBackOff

Cause:

App failing at startup
Wrong configs
